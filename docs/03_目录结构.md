# 目录结构与架构：如何读懂代码？

在复杂的软件工程中，把代码分门别类地放好，叫做“架构设计”。
本项目采用了非常经典的 **领域驱动设计 (DDD)** 结合 **分层架构**。即使你是小白，只要记住“职责分离”，就不会迷路。

## 📁 核心结构一览

```text
heal-mate-ai/
├── src/                    # 🚀 所有业务源代码都在这里
│   ├── app.py              # 🚪 程序的入口，运行 Streamlit 的起点
│   ├── ai_service.py       # 🤖 专门负责和 DeepSeek 大模型对话、生成提示词
│   ├── config.py           # ⚙️ 全局配置，如 API Key、AI 的基础设定
│   ├── database.py         # 🗄️ 数据库初始化和连接
│   ├── utils.py            # 🛠️ 公共小工具函数
│   │
│   ├── pages/              # 🎨 【UI 视图层】只管画界面
│   │   ├── home.py         # 首页与登录
│   │   ├── 1_consultation.py # 对话式问诊页面
│   │   └── 2_checkin.py    # 每日打卡页面
│   │
│   ├── services/           # 👔 【服务层】负责具体的业务逻辑
│   │   ├── auth_service.py    # 处理登录、注册逻辑
│   │   ├── plan_service.py    # 处理“找AI生成计划，并存入数据库”的流程
│   │   └── checkin_service.py # 处理打卡、计算连续天数逻辑
│   │
│   ├── repos/              # 💾 【数据访问层】只管写 SQL 读写数据库
│   │   ├── connection.py   # 获取数据库连接
│   │   ├── migrations.py   # 建表语句（如 CREATE TABLE ...）
│   │   ├── auth_repo.py    # 读写 users 和 sessions 表
│   │   └── plan_repo.py    # 读写 plans 表
│   │
│   ├── core/               # 🧠 【领域核心层】跨越各层的数据模型和状态
│   │   └── user_context.py # 定义用户处于什么状态（未开始、已填档案、已有计划）
│   │
│   └── ui/                 # 💄 【UI 组件层】
│       └── theme.py        # 存放统一的 CSS 样式和 Streamlit 主题配置
│
├── tests/                  # 🧪 【测试代码】
│   ├── conftest.py         # 测试环境准备（如创建一个临时空的数据库）
│   └── test_*.py           # 针对 services, repos, core 的具体测试用例
│
├── docs/                   # 📚 【文档目录】你正在看的地方
├── .github/workflows/      # 🤖 【CI/CD】自动化流水线（提交代码自动跑测试）
├── pytest.ini              # ⚙️ Pytest 测试框架配置文件
├── pyproject.toml          # ⚙️ Ruff 代码规范配置文件
└── requirements.txt        # 📦 依赖包清单 (pip install -r)
```

## 🔍 数据是怎么流动的？（以“打卡”为例）

假设用户在网页上点击了【完成打卡】按钮，代码的执行顺序是这样的：

1. **`pages/2_checkin.py` (前台)**：捕获到了按钮点击事件。它**不知道**怎么存数据库，所以它调用了 `checkin_service.save_checkin(...)`。
2. **`services/checkin_service.py` (大堂经理)**：收到请求后，它可能会先算一下用户连续打卡了几天，然后把整理好的数据，交给了 `checkin_repo.save_checkin(...)`。
3. **`repos/checkin_repo.py` (库管员)**：拿到数据后，写了一句 `INSERT INTO check_ins ...` 的 SQL 语句，把数据真正写到了 SQLite 硬盘文件里。

**为什么要这么麻烦？**
因为“解耦”。如果有一天我们想把 SQLite 换成 MySQL，我们**只需要**改 `repos/` 里的代码，`pages` 和 `services` 完全不用动！这就是好架构的魅力。
